## МОДУЛЬ 1. КОНТЕКСТ И ПРЕДПОСЫЛКИ 
 
### 1. ОГРАНИЧЕНИЯ СТАНДАРТНЫХ СРЕДСТВ 
 
**Что используют обычно:** 
- Apache Bench (ab) - простая программа для тестирования 
- curl - команда для одного запроса 
- JMeter - большая программа с графическим интерфейсом 
 
**В чем их проблемы:** 
 
**Apache Bench (ab):** 
- Может делать только одинаковые запросы подряд 
- Нельзя написать сценарий: "сначала зайди на главную, потом в каталог, потом в корзину" 
- Нет графиков, только цифры в консоли 
- Нельзя менять нагрузку во время теста 
- Пример: ab -n 1000 -c 10 http://site.com/ (1000 запросов, 10 одновременно, но все к одному адресу) 
 
**curl:** 
- Это вообще не инструмент для нагрузочного тестирования 
- Делает один запрос и показывает ответ 
- Можно написать скрипт на bash, который будет много раз вызывать curl, но это костыль 
 
**JMeter:** 
- Сложный интерфейс, нужно учиться работать с программой 
- Тесты хранятся в XML-файлах, их сложно читать и сравнивать в Git 
- Тяжелый, требует много памяти 
- Чтобы написать логику (если в ответе есть id, то использовать его), нужно разбираться в JMeter-функциях 
 
**Вывод:** Обычные инструменты хороши для простых тестов, но когда нужно моделировать реальное поведение пользователей - они неудобны. 
 
--- 
 
### 2. ПРОБЛЕМА, КОТОРУЮ РЕШАЕТ LOCUST 
 
Locust решает главную проблему: **разрыв между разработчиком и тестировщиком**. 
 
**Как было раньше:** 
1. Разработчик пишет код на Python 
2. Тестировщик учит JMeter, разбирается в XML 
3. Они говорят на разных языках, тесты сложно поддерживать 
 
**Как стало с Locust:** 
1. Разработчик (или тестировщик, знающий Python) пишет тест на Python 
2. Можно использовать все возможности языка: циклы, условия, функции, классы 
3. Тест хранится в Git, его можно ревьюить, как обычный код 
4. Нет необходимости учить новый инструмент с нуля 
 
**Какие задачи решает Locust:** 
 
🔹 **Сложные сценарии** - можно описать любой путь пользователя 
🔹 **Динамические данные** - можно достать ID из ответа и использовать в следующем запросе 
🔹 **Разные профили пользователей** - можно создать классы "Админ", "Гость", "Зарегистрированный" 
🔹 **Реалистичные паузы** - пользователь "думает" случайное время 
🔹 **Нагрузка в реальном времени** - видим графики прямо во время теста 
🔹 **Масштабирование** - можно запустить на сотне машин и создать миллион пользователей

### 3. ПОНЯТИЕ ИНЖЕНЕРНОЙ ГРАМОТНОСТИ В КОНТЕКСТЕ LOCUST 
 
**Что такое инженерная грамотность применительно к нагрузочному тестированию?** 
 
Это способность не просто запустить тест, а понимать: 
- Что именно мы тестируем 
- Какие метрики важны, а какие - нет 
- Где узкие места системы 
- Как интерпретировать результаты 
 
**Как Locust меняет мышление:** 
 
**До Locust (мышление "черный ящик"):** 
"Запущу-ка я 1000 запросов и посмотрю, сколько упадет" 
 
**После Locust (инженерное мышление):** 
"Создам модель поведения пользователя: 70%% времени он смотрит список, 30%% - читает конкретный пост. Между действиями думает 1-5 секунд. Посмотрю, как система справляется с таким профилем нагрузки." 
 
**Что значит "визуальная грамотность" в Locust:** 
 
Веб-интерфейс Locust показывает графики. Инженер должен уметь их читать: 
 
 **График RPS (Requests Per Second)** - растет ли он с увеличением пользователей? Если нет - значит система уперлась в потолок. 
 
 **График времени ответа** - где начинается резкий рост? Это точка, после которой система не справляется. 
 
 **Процентили (95%%, 99%%)** - если среднее время 100 мс, а 95%% запросов отвечают за 500 мс - это плохо. Значит, есть выбросы, которые портят опыт пользователей. 
 
**Вывод:** Locust учит мыслить как инженер, а не как "тестировщик-кнопкодав".

### 4. ПРИМЕНЕНИЕ В РЕАЛЬНЫХ ПРОЕКТАХ 
 
**Где и как используют Locust в промышленности:** 
 
 **Discord** (мессенджер для геймеров) 
  - Используют Locust для тестирования своих бэкендов перед крупными обновлениями 
  - Проверяют, выдержат ли серверы, когда миллионы пользователей одновременно заходят в голосовые каналы 
 
 **Microsoft** 
  - Тестируют некоторые облачные сервисы Azure 
  - Используют распределенный режим для создания огромной нагрузки 
 
 **SAP** 
  - Нагружают свои корпоративные системы 
  - Моделируют поведение тысяч сотрудников, работающих одновременно 
 
 **SendGrid** (сервис email-рассылок) 
  - Тестируют API для отправки писем 
  - Проверяют, сколько писем в секунду можно отправить без задержек 
 
 **IoT-компании** 
  - Нагружают MQTT-брокеры (протокол для умных устройств) 
  - Симулируют тысячи датчиков, отправляющих данные 
 
 **Финтех** 
  - Тестируют платежные системы 
  - Проверяют, что транзакции не теряются при пиковых нагрузках (Черная пятница, Новый год) 
 
**Пример из жизни:** 
Представьте, что онлайн-кинотеатр выпускает новый сезон популярного сериала. В первый вечер ожидается наплыв зрителей. Инженеры запускают Locust за неделю до премьеры, создают 100 000 виртуальных пользователей, которые ищут фильмы, ставят на паузу, перематывают. Если система падает - значит, нужно добавить серверов до премьеры.

## МОДУЛЬ 2. ОСНОВНЫЕ ИДЕИ И МЕХАНИЗМЫ 
 
### 5. ЦЕНТРАЛЬНЫЕ ОБЪЕКТЫ И АРХИТЕКТУРА 
 
Locust построен на нескольких ключевых понятиях: 
 
**HttpUser** - это класс, который представляет одного виртуального пользователя. У каждого пользователя есть: 
- Свои cookies и сессия 
- Свои переменные (например, список постов) 
- Свое поведение (какие задачи выполняет) 
 
**Как это работает под капотом:** 
Locust использует библиотеку **gevent**. Она позволяет запускать тысячи "зеленых потоков" (greenlet) в одном реальном потоке операционной системы. Это очень эффективно по памяти. 
 
**Процесс работы:** 
1. Вы запускаете Locust 
2. Locust создает экземпляры класса HttpUser (каждый в своем greenlet) 
3. Каждый пользователь в цикле выбирает случайную задачу (с учетом весов) 
4. Выполняет задачу 
5. Ждет указанное время 
6. Повторяет 
 
**Архитектура:** 
- **Мастер** - управляет тестом, собирает статистику, отдает веб-интерфейс 
- **Воркеры** - запускают пользователей, могут быть на разных машинах 
- **Веб-интерфейс** - показывает графики и позволяет управлять тестом

### 6. КЛЮЧЕВЫЕ МЕХАНИЗМЫ И ПРИЁМЫ РАБОТЫ 
 
**1. Декоратор @task** 
Помечает метод как задачу. Можно задать вес: 
```python 
@task(3)  # будет выполняться с весом 3 
def my_task(self): 
    pass 
``` 
 
**2. wait_time** 
Задает паузу между задачами: 
```python 
wait_time = between(1, 5)  # случайно от 1 до 5 секунд 
wait_time = constant(3)     # всегда 3 секунды 
wait_time = constant_pacing(10) # задача должна занимать 10 секунд 
``` 
 
**3. on_start и on_stop** 
Выполняются при старте и остановке пользователя: 
```python 
def on_start(self): 
    # авторизация, загрузка данных 
    self.client.post("/login", json={"user": "test"}) 
``` 
 
**4. HTTP-клиент** 
`self.client` - это как библиотека requests, но с автоматическим сбором статистики: 
```python 
response = self.client.get("/posts") 
response = self.client.post("/posts", json={"title": "New post"}) 
```

### 7. РАБОТА СО СТРУКТУРИРОВАННЫМИ ДАННЫМИ 
 
Locust позволяет легко работать с JSON, XML и другими форматами: 
 
**Получение и парсинг JSON:** 
```python 
response = self.client.get("/posts") 
posts = response.json()  # превращает JSON в список словарей 
first_post_id = posts[0]['id']  # берем ID первого поста 
``` 
 
**Отправка JSON:** 
```python 
self.client.post("/posts", json={ 
    "title": "Мой пост", 
    "body": "Текст поста", 
    "userId": 1 
}) 
``` 
 
**Сохранение данных между запросами:** 
```python 
def on_start(self): 
    self.user_id = 123  # сохраняем в self 
    self.posts = [] 
def some_task(self): 
    print(self.user_id)  # доступно здесь 
``` 
 
**Цепочки запросов (самое полезное!):** 
```python 
def create_and_view_post(self): 
    # 1. Создаем пост 
    create_resp = self.client.post("/posts", json={"title": "Test"}) 
    if create_resp.status_code == 201: 
        new_post_id = create_resp.json()['id'] 
        # 2. Сразу смотрим созданный пост 
        self.client.get(f"/posts/{new_post_id}") 
```

### 8. ИНТЕРАКТИВНОСТЬ И АВТОМАТИЗАЦИЯ 
 
**Веб-интерфейс (интерактивность):** 
- Запуск/остановка теста 
- Изменение количества пользователей на лету 
- Графики в реальном времени 
- Таблица со статистикой по каждому эндпоинту 
- Скачивание результатов в CSV 
 
**Режим без головы (автоматизация для CI/CD):** 
```bash 
locust -f locustfile.py --host=https://api.site.com --headless -u 100 -r 10 --run-time 1m --csv=results 
``` 
- `--headless` - без веб-интерфейса 
- `-u 100` - 100 пользователей 
- `-r 10` - 10 пользователей в секунду 
- `--run-time 1m` - тест длится 1 минуту 
- `--csv=results` - сохранить результаты в CSV 
 
**REST API Locust:** 
Можно управлять тестом через HTTP-запросы: 
```bash 
curl -X POST http://localhost:8089/swarm -d "user_count=100&spawn_rate=10" 
curl -X GET http://localhost:8089/stats/requests 
``` 
 
**Распределенный режим:** 
На мастере: 
```bash 
locust -f locustfile.py --master --host=https://api.site.com 
``` 
На воркерах (другие компьютеры): 
```bash 
locust -f locustfile.py --worker --master-host=192.168.1.100 
```
